# Manual Rule Explainer

`manual_rule_explainer` is a deterministic parse-graph builder driven by an explicit role spec.

It is a parallel path to the operation-level MDL/OR-Tools explainer. It does not generate operation candidates, does not score compression gain, and does not call OR-Tools. The existing operation explainer remains available.

## Input

The builder takes:

- `maskgen_explanation_evidence_v1`
- `maskgen_role_spec_v1`

The role spec is the only source of high-level semantic relations. Supported relation types are:

- `INSERTED_IN`
- `DIVIDES`
- `PARALLEL`

Unspecified label pairs are not promoted into high-level semantic relations by this module. Their faces fall back to residual unless they are referenced through another explicit rule.

## Output

The top-level output format is:

```json
{
  "format": "maskgen_manual_rule_explanation_v1",
  "explainer_profile": "manual_role_spec_parse_graph_v1",
  "selected_explanations": [],
  "generator_target": {
    "format": "maskgen_generator_target_v1",
    "target_type": "parse_graph",
    "size": [256, 256],
    "parse_graph": {
      "nodes": [],
      "relations": [],
      "residuals": []
    },
    "metadata": {}
  },
  "diagnostics": {},
  "validation": {}
}
```

The generator should train on `generator_target.parse_graph`. The rest is diagnostics and traceability.

## Construction Rules

`INSERTED_IN(subject_label, object_label)`:

- Builds a `support_region` for matched object-label faces.
- Builds an `insert_object_group` for the subject label.
- Builds one `insert_object` per matched subject-label face.
- Emits `inserted_in` and `contains` relations.
- If an insert face matches multiple support labels, the support label with the largest shared boundary wins.

`DIVIDES(subject_label, object_label)`:

- Builds a `divider_region` for subject-label faces adjacent to object-label faces.
- Builds a `support_region` for matched object-label faces.
- Emits a `divides` relation.

`PARALLEL(subject_label, object_label)`:

- Builds `support_region` nodes for both matched labels.
- Emits an `adjacent_to` relation.

Any face not referenced by an explicit rule becomes a `residual_region`.

## Ownership Model

The direct builder separates ownership from context:

- `owned_face_ids`: faces whose geometry is generated by this node.
- `referenced_face_ids`: faces only used as grouping or relation context.
- `face_ids`: compatibility alias for owned faces on geometry nodes.

Every evidence face must have exactly one owner. Insert groups do not own their child faces; they only reference them. This avoids the earlier ambiguity where the same building face could be owned by both `insert_object_group` and `insert_object`.

When role rules conflict, the current fixed priority is:

```text
divider subject face / insert subject face
  > support context face
  > residual fallback
```

For example, if a water face is explicitly `INSERTED_IN` a plain region and also appears in a weak `PARALLEL` relation, the water face is owned by the `insert_object`, not by a support node. The skipped support relation is visible through diagnostics and residual/ownership counts.

Validation records:

- `all_faces_owned_exactly_once`
- `duplicate_owned_face_count`
- `unowned_face_count`
- `referenced_face_count`

## Hierarchical Relation Endpoints

Relations can target either geometry nodes or group nodes.

This is the minimal support for chained structures such as:

```text
A is parallel to B
C is inserted in B
D divides the group of C objects
```

The builder does not hard-code any dataset-specific chain. It follows the explicit role spec and resolves relation endpoints by label:

```text
DIVIDES subject:
  divider node -> insert group -> support node

DIVIDES object / PARALLEL side:
  support node -> insert group -> divider node
```

Therefore if a label already has an `insert_object_group`, later rules can reference that group directly. For example, `DIVIDES(road, building)` can produce:

```json
{
  "type": "divides",
  "divider": "divider_0",
  "support": "insert_group_0"
}
```

This keeps geometry ownership strict: each building face is still owned by its own `insert_object`, while the group is only a relation endpoint and context node.

## Important Difference From OR-Tools Explainer

The operation explainer uses candidates, costs, compression gain, and a global selector. This module does not.

That means:

- There is no "selected vs not selected" ambiguity.
- There is no automatic role inference.
- There is no optimization objective.
- The output is only as complete as the manual role spec and local adjacency evidence.

The tradeoff is that the same evidence face may be referenced by more than one conceptual node when the role spec says a label participates in multiple relative roles. This is recorded in `diagnostics.duplicate_evidence_face_ids`.

## Command

```bash
python scripts/build_manual_rule_explanation_single.py \
  --evidence-json outputs/visualizations/explanation_evidence_val37_initial_micro_absorbed.json \
  --role-spec configs/role_specs/remote_256.json \
  --output outputs/visualizations/manual_rule_val37.json
```

The same path is also available through the operation script without changing its default behavior:

```bash
python scripts/build_operation_explanation_single.py \
  --mode manual_rule \
  --evidence-json outputs/visualizations/explanation_evidence_val37_initial_micro_absorbed.json \
  --role-spec configs/role_specs/remote_256.json \
  --output outputs/visualizations/manual_rule_val37.json
```

Optional flags:

- `--include-all-faces-of-support-labels`
- `--include-all-faces-of-divider-labels`
- `--min-shared-length`

## Current Limits

- No renderer is implemented here; `render_iou` remains `null`.
- It does not discover missing relations.
- It does not resolve semantic correctness beyond the explicit role spec.
- It does not remove the older operation-level explainer.
